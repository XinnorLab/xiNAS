## Purpose

This plan describes how to implement xiNAS Phase 0 as a single-node Control Path foundation. Phase 0 wraps the existing xiNAS provisioning, xiRAID, XFS, NFS, RDMA network, configuration history, diagnostics, TUI, and MCP components behind a typed API, local agent, local state store, task engine, audit trail, and automation-safe plan/apply model.

Phase 0 must not implement clustered HA, Active/Passive failover, Active/Active NFS, witness quorum, Pacemaker orchestration, or cross-node NFS lock recovery. It must prepare the architecture so those capabilities can be added later without redesigning the control model.

---

## Delivery principles

1. Preserve the existing Data Path.
   - NFS I/O, xiRAID, XFS, RDMA networking, and tuned kernel/NFS configuration must continue to work without depending on the new API during steady state.
   - Restarting Control Path services must not interrupt existing client I/O.

2. Use the current xiNAS implementation as bootstrap, not as final day-2 control.
   - Ansible remains the installer/provisioning mechanism.
   - Day-2 operations move to API tasks.
   - Shell scripts remain compatibility wrappers only.

3. Introduce one source of operational truth.
   - Desired state and observed state are separate.
   - All clients — TUI, CLI, MCP, automation — consume the same API.
   - Safety decisions live in the Control API / agent, not in each UI.

4. Implement plan/apply everywhere before broad write support.
   - Every mutating operation must first produce a deterministic plan.
   - Apply must be tied to a plan revision and idempotency key.
   - Unsafe changes must show blast radius and blockers.

5. Build Phase 0 as a migration layer.
   - It must import current xiNAS state from real system configuration.
   - It must not require a clean reinstall to become manageable.
   - Existing arrays, filesystems, exports, netplan files, and config-history snapshots must be discoverable.

---

## Target Phase 0 architecture

The target Phase 0 implementation contains the following runtime components:

- `xinas-api`: local versioned REST API, for example `/api/v1`.
- `xinas-agent`: privileged local executor and observer.
- Local state store: single-member etcd, SQLite, or embedded KV store with future-compatible key layout.
- Task engine: durable operation manager for plan/apply, locks, logs, status, and recovery hints.
- `xinas-nfs-helper`: structured NFS export/session/quota helper behind the agent/API.
- Storage adapter: xiRAID integration, preferably through `xiraid-server` gRPC where available.
- Filesystem adapter: XFS discovery, mount, create/import, grow, quota, and dependency checks.
- Network adapter: RDMA/IB netplan rendering and validation.
- Config-history bridge: snapshots, diffs, rollback metadata, drift detection.
- `xinasctl`: machine-oriented CLI with JSON output.
- Updated `xinas-menu`: TUI client of the Control API.
- `xinas-mcp`: Phase 0 MCP client of the Control API, read-only and plan-only by default.
- Health/support subsystem: unified health profiles and support bundle generation.
- RBAC/audit subsystem: local roles, append-only audit records, correlation IDs.

## Workstream 0 — Architecture freeze and backlog decomposition

### Objective

Convert Phase 0 requirements into engineering-ready contracts, repositories, ownership boundaries, and testable milestones.

### Tasks

1. Freeze Phase 0 scope.
   - Explicitly mark HA, Pacemaker, witness, floating IP failover, clustered etcd, NFSv4 lock reclaim across nodes, Active/Active, SMB, and S3 as out of scope.
   - Mark their required data-model placeholders as in scope where needed.

2. Create the Phase 0 architecture decision records.
   - State store choice: embedded SQLite/KV vs single-member etcd.
   - API style: REST only, or REST plus gRPC internal calls.
   - Task engine persistence model.
   - Agent privilege model.
   - RBAC/audit model.
   - Packaging model: systemd services first, containers optional.

3. Define component ownership.
   - API/state/task team.
   - Agent/storage/filesystem team.
   - NFS/network team.
   - TUI/CLI/MCP team.
   - QA/CI/release team.
   - Documentation/support team.

4. Split requirements into implementation epics.
   - Control API.
   - State store.
   - Agent.
   - Task engine.
   - xiRAID adapter.
   - Filesystem adapter.
   - NFS helper integration.
   - Network adapter.
   - Config history.
   - Health/support.
   - Security/audit.
   - UI/CLI/MCP.
   - Installer/upgrade.
   - Test automation.

### Exit criteria

- Phase 0 scope document is approved.
- API object list is approved.
- State model is approved.
- Initial backlog is estimated.
- CI gates are defined before implementation starts.

## Workstream 3 — xinas-agent

### Objective

Implement the local privileged observer/executor used by the API.

### Tasks

1. Implement agent identity.
   - Generate UUIDv4 `controller_id`.
   - Store under `/etc/xinas/controller-id` or equivalent.
   - Preserve across reboot and upgrade.

2. Implement read/observe methods.
   - System inventory.
   - Disk/NVMe inventory.
   - xiRAID arrays.
   - Filesystem state.
   - Mount state.
   - NFS exports.
   - NFS service state.
   - NFS client/session visibility where available.
   - Network/RDMA state.
   - Netplan files.
   - Systemd units.
   - Relevant logs.

3. Implement typed execution methods.
   - RAID lifecycle operations.
   - Filesystem lifecycle operations.
   - NFS export operations through helper.
   - NFS profile changes.
   - Network changes.
   - Service restart/reload where explicitly allowed.
   - Health checks.
   - Support bundle collection.

4. Implement dependency guardrails.
   - Share → filesystem → array → disks.
   - NFS profile → exports → clients.
   - Network interface → service IPs → client mount profiles.
   - Mounted filesystem → active NFS export → active clients/locks.

5. Implement execution sandbox.
   - No arbitrary shell execution from API/TUI/CLI/MCP.
   - Only typed operations.
   - All subprocess use isolated inside approved adapters/helpers.
   - Structured stdout/stderr capture for task logs.

6. Implement agent heartbeat and readiness.
   - API can report agent offline/degraded/healthy.
   - Read-only cached state remains available when agent is offline.
   - Mutating operations fail cleanly if executor is unavailable.

### Exit criteria

- Agent reports complete observed state.
- Agent executes typed operations only.
- Disabling agent blocks writes but not cached reads.
- Agent restart does not lose controller identity.
- Unsafe destructive operations are blocked.

## Workstream 4 — Task engine and operation manager

### Objective

Make every mutating operation durable, observable, idempotent, and safe.

### Tasks

1. Implement task lifecycle.
   - `queued`
   - `running`
   - `success`
   - `failed`
   - `cancelled`
   - `requires_manual_recovery`

2. Implement task persistence.
   - Task metadata.
   - Stage logs.
   - Input hash.
   - Plan hash.
   - Result hash.
   - Final observed state.
   - Error details.
   - Remediation hints.

3. Implement plan/apply binding.
   - `plan` produces `plan_id`.
   - `apply` requires valid `plan_id`.
   - `apply` validates expected revision.
   - `apply` refuses stale plan.

4. Implement idempotency.
   - Client-provided `idempotency_key`.
   - Retry returns original task.
   - Duplicate applies are prevented.

5. Implement resource locks.
   - Disk locks.
   - Array locks.
   - Filesystem locks.
   - Share locks.
   - NFS profile lock.
   - Network interface lock.

6. Implement cancellation rules.
   - Preflight can be cancelled.
   - Reversible stages can be cancelled.
   - Non-interruptible stages return explicit refusal.

7. Implement failure recovery states.
   - Failed before change.
   - Failed after partial change with rollback available.
   - Failed after partial change with manual recovery required.
   - Failed after successful system change but state-store update failed.

### Exit criteria

- Long-running task survives API restart.
- Conflicting operations are blocked.
- Retry does not duplicate operations.
- Failed task has actionable diagnostics.
- Cancellation behavior is deterministic.

## Workstream 8 — Network/RDMA adapter

### Objective

Centralize xiNAS-managed IB/RDMA network configuration and make it safe for day-2 operations.

### Tasks

1. Implement network discovery.
   - Interfaces.
   - Driver.
   - RDMA capability.
   - Link state.
   - MTU.
   - IP addresses.
   - Routes.
   - PBR rules.
   - Netplan file ownership.

2. Implement canonical netplan rendering.
   - xiNAS owns `/etc/netplan/99-xinas.yaml` for IB/RDMA interfaces.
   - Management Ethernet remains outside xiNAS unless explicitly supported.
   - Render IPs, MTU, PBR, and interface state.

3. Implement cross-file conflict detection.
   - Detect IB/RDMA definitions in non-xiNAS netplan files.
   - Report as blocker or auto-clean only when explicitly planned.
   - Detect duplicate IPs and stale PBR.

4. Implement flush-before-apply flow.
   - Flush PBR rules in xiNAS-managed range.
   - Flush routes in xiNAS-managed tables.
   - Flush IPs on managed RDMA interfaces.
   - Render `99-xinas.yaml`.
   - Run `netplan generate`.
   - Apply only after validation.

5. Implement network plan/apply.
   - Add/change/remove interface IP.
   - Add/change/remove IP pool.
   - Change MTU.
   - Enable/disable interface where supported.
   - Validate client impact.

6. Add tests.
   - Duplicate netplan conflict.
   - Stale IP removal.
   - PBR route cleanup.
   - Invalid CIDR.
   - Interface name mismatch.
   - NFS-RDMA mount after network apply.

### Exit criteria

- Network state is visible through API.
- `99-xinas.yaml` is canonical for managed RDMA interfaces.
- Duplicate netplan definitions are detected.
- Network changes are plan/apply tasks.
- Existing client I/O impact is clearly reported before apply.

## Main engineering risks and mitigations

### Risk 1 — Too much logic remains in shell scripts

Mitigation:
- Add CI checks that prevent new day-2 features from being implemented only in shell.
- Keep shell scripts as wrappers around `xinasctl` or API.

### Risk 2 — API and agent duplicate safety logic

Mitigation:
- API owns policy/preflight orchestration.
- Agent owns low-level evidence and typed execution.
- Clients own display only.

### Risk 3 — State store diverges from real system

Mitigation:
- Keep desired and observed state separate.
- Run drift checks on schedule and after every task.
- Surface drift in health and task preflight.

### Risk 4 — Network changes break management access

Mitigation:
- Keep management Ethernet outside xiNAS-controlled RDMA config in Phase 0.
- Apply xiNAS management only to RDMA/IB interfaces unless explicitly supported.
- Require plan output with client impact.
- Use `netplan generate` and conflict detection before apply.

### Risk 5 — Storage operations become destructive by accident

Mitigation:
- Require dependency traversal.
- Require explicit dangerous flag.
- Require plan/apply with expected revision.
- Block delete when mounted/exported/active.
- Never delete data on share delete by default.

### Risk 6 — MCP bypasses safety model

Mitigation:
- MCP calls Control API only.
- Default MCP mode is read-only and plan-only.
- No direct subprocess execution from MCP.
- Audit every MCP tool call through API task/audit IDs.

### Risk 7 — Existing-node upgrade is unreliable

Mitigation:
- Implement import-first upgrade.
- Generate desired state from current observed system.
- Do not apply changes during import unless explicitly requested.
- Run upgrade tests on real previous xiNAS installations.
